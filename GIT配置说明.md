# CleanC - Git配置说明

本文档说明了本项目的Git版本控制配置，以及如何正确地使用Git进行协作开发。

## 忽略的文件和目录

项目中的`.gitignore`文件配置了不需要提交到代码仓库的文件和目录。这些包括：

1. **依赖包文件夹** - `node_modules/`
   - 这个目录包含所有npm安装的依赖包，通常很大且不需要版本控制
   - 团队成员应该通过运行`npm install`自行安装依赖

2. **构建和打包输出** - `dist/`, `out/`, `build/`, `release/`
   - 这些目录包含构建和打包过程生成的文件
   - 这些文件可以通过源代码重新生成，不需要版本控制

3. **运行时生成的文件** - 日志文件、临时文件、测试结果等
   - 这些文件是在程序运行或测试过程中生成的，每个开发者会有自己的版本

4. **敏感信息** - 环境变量文件、密钥等
   - 这些文件包含敏感信息，不应该提交到公共仓库

5. **打包后的应用程序** - `.exe`, `.msi`等
   - 这些是最终的安装程序或可执行文件，通常体积较大且可以通过源代码重新生成

## 提交代码的最佳实践

1. **频繁提交**
   - 每个逻辑更改都应该单独提交，避免在一个提交中混合多个功能或修复

2. **清晰的提交消息**
   - 提交消息应该简洁明了地描述所做的更改
   - 建议格式：`类型(范围): 描述`，例如`feat(卸载): 添加程序卸载功能`

3. **使用分支**
   - 为每个新功能或修复创建一个新分支
   - 完成后通过拉取请求(Pull Request)合并到主分支

4. **代码审查**
   - 所有代码更改都应该经过至少一名团队成员的审查
   - 审查应该关注代码质量、可维护性和测试覆盖率

## 常用Git命令

```bash
# 克隆仓库
git clone <仓库URL>

# 创建并切换到新分支
git checkout -b <分支名>

# 查看更改状态
git status

# 添加更改到暂存区
git add <文件或目录>

# 提交更改
git commit -m "提交消息"

# 推送到远程仓库
git push origin <分支名>

# 从远程仓库拉取更新
git pull origin <分支名>

# 合并另一个分支的更改
git merge <分支名>
```

## 特别说明

- 对于`package-lock.json`文件，我们选择将其纳入版本控制，以确保所有开发者使用相同版本的依赖。
- 如需添加新的忽略规则，请更新`.gitignore`文件并通知所有团队成员。
- 不要使用Git提交大文件，尤其是二进制文件和可执行文件。如需共享这类文件，请使用其他文件共享方式。
